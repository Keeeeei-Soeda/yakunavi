# 薬剤師：応募済み求人の表示・操作の改善見通し

「応募済みの案件に再度応募しようとしたとき」の UX をどうするかの見通しです。**通常であれば「応募済み」と出すのがよい**という結論と、その実装方針をまとめています。

---

## 1. 推奨方針：「応募済み」を出す

**はい。通常であれば「応募済み」と出して、二重応募を防ぐのがよいです。**

理由の要点だけ挙げます。

- **誤操作の防止** … 同じ求人に何度も応募ボタンを押してエラーになる体験を避けられる。
- **状態の明示** … 「すでに応募している」ことが一目で分かり、応募管理との一貫性が取れる。
- **導線の整理** … 応募済みなら「応募する」ではなく「応募内容を確認する」など、適切な次のアクションに誘導できる。
- **業界慣行** … 求人サイトでは「応募済み」バッジ・文言で同様の扱いが一般的。

したがって、**一覧・詳細の両方で「応募済み」を出し、応募ボタンの扱いを変える**改善を推奨します。

---

## 2. 改善のレベル別パターン

| レベル | 一覧 | 詳細 | コメント |
|--------|------|------|----------|
| **A. 最小** | 変更なし | 応募済みなら「応募済み」表示＋ボタン無効 or 非表示 | 詳細だけ直す。一覧は現状のまま。 |
| **B. 推奨** | 応募済みに「応募済み」バッジ／ラベル。クリックは詳細へ可。詳細では「応募済み」＋応募ボタンは出さない or 「応募内容を確認」へ | 一覧でも状態が分かり、詳細で二重応募の導線をなくせる。 |
| **C. 強め** | 応募済みは一覧から非表示（フィルターで「未応募のみ」など） | ニーズ次第。応募済みを隠すと「もう応募した一覧」が見づらくなるので、オプション程度が無難。 |

**おすすめは B** です。「応募済み」を明示しつつ、詳細には行けるが「もう一度応募」はできないようにします。

---

## 3. 実装で必要なこと（見通し）

### 3.1 データ

- **「この薬剤師が応募済みの求人ID一覧」** があれば足りる。
- 既存 API: **`GET /api/applications/pharmacist/:pharmacistId`** で応募一覧が取れるので、  
  `application.jobPostingId` を集めた **`Set<number>`（または配列）** をフロントで持つ。

追加 API は必須ではなく、既存の応募一覧から「応募済み jobPostingId」を導く形でよいです。

### 3.2 求人一覧（`/pharmacist/jobs`）

| 項目 | 内容 |
|------|------|
| 取得 | ページ表示時（または検索時）に、上記 API で応募一覧を取得し、**応募済みの jobPostingId の Set** を保持。 |
| 表示 | 各カードで `appliedJobIds.has(job.id)` なら **「応募済み」バッジ／ラベル** を表示（例: 青系バッジ）。 |
| ボタン | 「詳細を見る」は **そのままクリック可**（詳細へ遷移）。  
  または「応募済み」のときは「応募内容を確認」と文言を変え、`/pharmacist/applications` や該当応募詳細へ飛ばす、でも可。 |
| クリック | 応募済みでも **詳細ページへは遷移してよい**（情報確認・応募内容確認のため）。 |

### 3.3 求人詳細（`/pharmacist/jobs/[id]`）

| 項目 | 内容 |
|------|------|
| 取得 | ページ表示時に、**この薬剤師がこの求人に応募済みか**を判定する必要がある。  
  - 案1: 一覧と同様に応募一覧 API を呼び、`jobPostingId === jobId` の有無で判定。  
  - 案2: 「この求人への応募有無」専用の軽い API を用意（あれば実装がすっきりするが、必須ではない）。 |
| 表示 | 応募済みなら **「この求人には応募済みです」** のようなメッセージを上部や CTA 近くに表示。 |
| ボタン | **「この求人に応募する」は出さない**（非表示）、または **無効化して「応募済み」と表示**。  
  代わりに **「応募内容を確認する」** などで `/pharmacist/applications` や該当応募詳細へリンクする。 |
| モーダル | 応募済みの場合は **応募フォームのモーダルは開かない**（ボタン自体を出さない or 無効化で十分）。 |

### 3.4 バックエンド

- **現状の重複応募チェックはそのまま維持**（最終防衛として必須）。
- 必要なら「この薬剤師がこの求人に応募済みか」を返す **GET** を 1 本追加すると、詳細ページの判定が楽になる（任意）。

---

## 4. 文言・UI の例

- **一覧**  
  - 応募済みカード: タイトル横 or カード内に **「応募済み」** バッジ（青系など）。  
  - ボタン: 「詳細を見る」のまま、または「応募内容を確認」。
- **詳細**  
  - 応募済み時: 「この求人にはすでに応募済みです。応募状況は応募管理からご確認ください。」＋「応募管理へ」リンク。  
  - 「この求人に応募する」ボタンは非表示、または disabled＋「応募済み」。

---

## 5. まとめ

- **「通常であれば応募済みと出すのがいいか」→ はい。一覧・詳細の両方で「応募済み」を出し、詳細では応募ボタンを出さない（または無効＋応募管理へ誘導）のがよいです。**
- 実装は「応募一覧 API で応募済み jobPostingId を取得 → 一覧でバッジ表示、詳細で応募ボタン制御」が中心で、既存 API で足りる想定です。
- バックエンドの二重応募チェックは残しつつ、フロントで「応募済み」を明示することで、エラーに頼らない分かりやすい UX にできます。
